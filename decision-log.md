# Winkidoo – Decision Log

Key architectural and product decisions. Keep updated so we don’t repeat mistakes or lose context.

---

## 2026-02-23

- **Evolved concept:** Winkidoo is a "Digital Surprise Vault" for couples, not only hidden text. Unlock can be any digital surprise (text MVP1; later photo, voice, etc.). AI judge is a character with multiple personas.
- **MVP1 scope:** Text-only surprises. Both unlock methods: Persuade (one convinces judge) and Collaborate (both submit). All 5 judge personas in MVP1. Dynamic win threshold = persona base + difficulty × 5. Google Gemini Flash for judge. Full Winks economy from day one. Dark mysterious aesthetic. Privacy: client-side encryption + auto-delete. Accessibility: semantics and system text scaling.
- **Stack:** Flutter 3.22+, Supabase (Auth, Realtime, Postgres), Riverpod 2.x (no code gen for now), Gemini Flash, encrypt package for AES.
- **Models:** Plain Dart classes with fromJson/toJson (no Freezed code gen in repo) for Surprise, Attempt, Couple, JudgeResponse, WinksBalance.
- **Encryption:** Key derived from couple_id so both partners can decrypt. Server only stores ciphertext. True E2E with shared secret can be added later.
- **Winks:** New users get 10 Winks (row created on first balance read). Free tier: 3 attempts/day. Extra attempt costs 1 Wink. Hint 5 Winks, instant unlock 50 Winks (UI later).
- **Realtime:** Supabase Realtime channel on `surprises` filtered by `couple_id`; on event we invalidate surprisesListProvider so vault list refreshes.
- **Schema:** `couples`, `surprises`, `attempts`, `winks_balance`, `transactions`. RLS so users only see their couple’s data and own winks/transactions.
- **Supabase config at runtime:** Never initialize Supabase with empty URL/key. In `main.dart`, check `String.fromEnvironment('SUPABASE_URL'/'SUPABASE_ANON_KEY')` and if empty show `ConfigErrorApp`; otherwise initialize. Local dev may use default values in code; production should use `--dart-define` and not commit keys.
- **Couple provider robustness:** Supabase can return `[]` or non-Map for “no couple” / single-row queries. In `couple_provider.dart` only parse when result `is Map<String, dynamic>`; wrap in try/catch and return `null` on error so UI shows link screen, not “Something went wrong”.
- **Join-by-code robustness:** Join flow uses `.maybeSingle()`; response may be List or other type on some platforms. In `couple_link_screen.dart` treat result as valid only when `raw is Map<String, dynamic>`, then use `res['id']` for the update.
- **Testing couple link:** Use two different Supabase accounts (two emails) to test create vs join; same account on two devices will consume the code (user_b_id set to same user) and show “already used” on retry.
- **Post-join provider hardening:** `surprisesListProvider` and `surpriseByIdProvider` accept List or single Map from Supabase and parse only Map elements; try/catch returns [] or null. `winksBalanceProvider` checks `res is Map<String, dynamic>` and uses try/catch so vault screen never throws after join (avoids “Something went wrong” on macOS when API shape differs).
- **AI Judge pro responses:** Use Gemini `responseSchema` (Dart `Schema.object` with `requiredProperties: ['commentary']`) so the model always returns a structured object with commentary. Retry once when the first response is valid JSON but commentary is empty/short; use a short follow-up prompt asking for in-character commentary. Use a small list of varied fallback messages (random pick) instead of a single repeated line when commentary is still missing after retry.
- **Judge "how to impress" answers:** When the seeker or creator asks what they should do to impress the Judge or how to win, the Judge must answer with concrete, persona- and context-aware guidance (1–3 ideas), not refuse or repeat "the magic comes from you" without suggestions. Implemented via: (1) explicit prompt rule, (2) per-persona "what this judge wants" map (`_howToImpressByPersona`), (3) unlock-method line from `surpriseContextHint` (persuade vs collaborate), (4) optional `howToImpressHint` parameter on `judgeChat()` for future game_mode/surprise_type (e.g. Date Night, Sing-Off). Added repetition rule: vary reactions and give new angles instead of repeating the same phrase.
- **Web quote detection:** Handle "sounds like from the web" (generic quotes, famous lines, search-result phrasing) via prompt only: new `webQuoteRule` instructs the Judge to respond in character with a warm, witty nudge that encourages original thinking (e.g. "little help from the internet", "tap into your own brain") without shaming. Two in-prompt examples (Sassy Cupid and Poetic Romantic) set the tone. No schema or UI change; commentary-only behavior.
- **Judge opener variation:** Avoid repeating "Oh honey" (or same pet name) every message. New `openerRule` in judge prompt: switch opening words (bestie, sweetheart, love, darling, or no opener). Sassy Cupid persona explicitly says to vary openers and never use the same opener two messages in a row. In-prompt examples updated to show varied openers so the model doesn’t default to one phrase.

## 2026-02-25

- **Wink+ subscription:** Couple-level premium via `couples.wink_plus_until` (timestamptz). When now() < wink_plus_until, couple has Wink+. Benefits: 10 free judge attempts/day (vs 3), access to all 5 judge personas (Chaos Gremlin, The Ex, Dr. Love gated in Create Surprise). No IAP yet — placeholder “Coming soon” on Wink+ screen; backend or admin can set wink_plus_until for testing or future Stripe/IAP integration.
- **Effective free attempts:** `effectiveFreeAttemptsPerDayProvider` (winks_provider) derives from couple; submission screen uses it so Wink+ users get 10 free attempts before spending Winks.
- **Hint and instant-unlock UI:** Already present on reveal and battle chat; added Semantics and use AppConstants for cost labels (5 and 50 Winks).
- **Phase 1 — Responsive UI:** LayoutBuilder + MediaQuery; desktop breakpoint 700px (app_constants). ResponsiveVaultShell: web desktop = sidebar vault (320px) + main Navigator; mobile = bottom nav + FAB. Vault list uses desktopDetailNavigatorKey to push battle/create on desktop into main area. kIsWeb used to avoid desktop-only APIs on web.
- **Phase 1 — Dual theme:** Light theme "Blush & Wink" in app_theme.dart (lightBackgroundStart/End, lightSurface, lightTextPrimary/Secondary). ThemeModeNotifier persists theme_mode (system/light/dark) in SharedPreferences. App uses themeModeProvider; all gradient backgrounds use AppTheme.gradientColors(Theme.of(context).brightness). Vault header has icon toggle to cycle system → light → dark.
- **Phase 1 — Micro-animations and haptics:** Reveal: HapticFeedback.mediumImpact() on unlock when !kIsWeb. Battle send button: ScaleTransition (0.88 on tap) + HapticFeedback.lightImpact() on send when !kIsWeb.
- **Phase 1 — Onboarding:** onboarding_provider (onboardingCompleteProvider, createFirstSurprisePromptSeenProvider) persisted. OnboardingScreen: 3-page PageView (value prop, couple link, first surprise); app shows it once after login if !onboardingComplete. Empty vault: EmptyVaultState with CTA "Create your first surprise".
- **Phase 1 — Surprise types:** Migration 004 adds surprise_type (text/photo/voice) and content_storage_path. Surprise model: surpriseType, contentStoragePath, isPhoto, isVoice. Storage bucket `surprises`; photo paths coupleId/id.jpg, voice coupleId/id.m4a. Create: type selector; photo via image_picker → upload; voice via record package → temp .m4a → upload. Reveal: photo = signed URL + Image.network; voice = signed URL + audioplayers UrlSource.
- **Phase 1 — Quality:** ErrorScreen (message, onRetry, onBack) for auth/couple errors. Vault loading: 4 SkeletonCards; error state has Retry (invalidate surprisesListProvider). Tests: app smoke (WinkidooApp), Surprise.fromJson (text + photo). PWA: web manifest + index.html title/description.
- **Smoke-test polish — Error screens:** Vault list and battle chat (messages + full-screen) use ErrorScreen with friendly copy and onRetry/onBack instead of raw error text.
- **Smoke-test polish — Semantics:** Submission “Submit to judge” (Send submission to judge); battle chat send (Send message to judge); create surprise type selector + Lock it (Create surprise) for TalkBack/VoiceOver.
- **Smoke-test polish — Skeletons:** SkeletonMessageRow widget; battle chat loading (initial and messages) shows skeleton rows instead of spinners.
- **Smoke-test polish — Copy:** ErrorScreen default “Something went wrong. Try again?”; judge prompt adds commentary tone (witty, warm, romantic, playful roasts).

## 2026-02-26

- **Realtime sync layer (before UI polish):** Battle screen must reflect live surprise state and navigate to reveal exactly once on resolve (local or partner). Implemented: (1) Migration 008 adds `surprises` to `supabase_realtime` so row UPDATEs are delivered. (2) BattleRealtimeService: same channel subscribes to both `battle_messages` (existing) and `surprises` (filter id = surpriseId, UPDATE only); optional `onSurpriseChanged(PostgresChangePayload)` callback. (3) Provider auto-update: on surprise row callback, invalidate `surpriseByIdProvider(surpriseId)`, `surprisesListProvider`, `battleMessagesProvider` so next ref.watch refetches; no provider type change. (4) Safe auto-navigation: when `newRecord['battle_status'] == 'resolved'`, if `!_navigatedToVerdict` and current route is still battle, set guard, fetch verdict message, build JudgeResponse from it, get creatorId from payload, then context.go to reveal. (5) Prevent double navigation: single guard `_navigatedToVerdict` set true before context.go in local seeker-win path, instant-unlock path, and realtime path; verdict-in-messages path already set it. Ensures animations and UI polish later use correct state.
- **Push notifications (Firebase/FCM):** Migrations 009 (`user_push_tokens` table, RLS) and 010 (multi-device: `id` PK, `push_token` unique). Firebase project with Android, iOS, Web apps; `google-services.json` and `GoogleService-Info.plist` gitignored; web config in `web/index.html`. `push_service.dart` upserts with `onConflict: 'push_token'`. Edge Function `send_battle_notification` sends only on relevant field changes; idempotency and rate-limit note. See **docs/FIREBASE_AND_PUSH_SETUP.md** for deploy/webhook/secret checklist.
- **Documentation and memory:** README Development Log and project structure updated; decision-log and LOCAL_VALIDATION.md reflect current state. Next: local run, LOCAL_VALIDATION checklist, Part 1 manual smoke-test.
- **CoupleLinkScreen loading fix:** App was sometimes stuck on a full-screen pink spinner (CoupleLinkScreen). Two fixes: (1) When couple is already linked, explicitly `context.go('/shell/vault')` in a post-frame callback so we don’t rely only on router redirect. (2) Couple-link loading state is no longer infinite: after 8s show “Taking too long? Check your connection” and a Retry button that invalidates `coupleProvider`. Ensures users never hang forever on that screen.
- **Data-driven judges:** Judges are now DB-driven (migration 011). Table `judges` extended with tagline, difficulty_level, chaos_level, tone_tags, preview_quotes, primary_color_hex, created_at, is_premium; season_start/season_end timestamptz. Active list = permanent (season null) or now() between season window; lock = is_premium from DB. `activeJudgesProvider` fetches and filters in client; `judgeByPersonaIdProvider(personaId)` for battle/reveal (no season filter so existing surprises still resolve). JudgeSelectionScreen watches activeJudgesProvider (loading/empty); seasonal badge on portrait ("Limited Time" / "Ends in X days" when ≤3 days). Judge.selectionList and Judge.forPersonaId removed; Judge.placeholder(personaId) for fallback when DB lookup fails.
- **New Judge badge:** `judges.is_new` column (migration 012); show "New" badge for seasonal judges when `is_new` is true and within 7 days of season start or created_at; client hides badge after 7 days (no scheduled job).
- **Season-launch push:** When a new seasonal judge becomes active (season_start <= now, is_new true, season_push_sent false), Edge Function sends one-time "✨ A New Judge Has Arrived" to all users with push tokens; then sets judges.season_push_sent = true. Deep link type season_launch opens /shell/create (judge selection). Same send_battle_notification function; add Database Webhook on public.judges (INSERT, UPDATE). Future refinement: filter tokens by updated_at last 90 days, or only users in a couple, or paginate sends to avoid heavy full-table fetch at scale.
